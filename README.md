## The problem

AI can write code faster than any team can review it. But speed without trust is a
liability: a diff that "looks fine" can leak secrets, pull in a vulnerable dependency,
regress performance, or drop an irreversible migration into production. Whether a change
is written by a human or generated by a model, the hard question is the same —

> **Can I trust this change, and can I prove it?**

## The solution

ProofForge treats every change as a **Proof-Carrying Change**: no modification is
considered trustworthy without verifiable technical evidence attached to it.

For a pull request, ProofForge runs the repository's tests inside an **isolated
sandbox**, collects security and dependency evidence, computes a **transparent risk
score**, applies **configurable policies**, and emits a hashable
**`proof-manifest.json`** that any other machine can independently verify.

## What "verifiable" means here

Three properties do the work, and each is enforced rather than asserted.

**Evidence records who measured it.** Every manifest carries a `collectors[]` list saying
which collector ran and what happened to it. This exists because a `0` is ambiguous — zero
vulnerabilities found and no scanner installed produce the same number, and only one of
them is good news. A collector that did not run is reported as `unavailable`, the risk
score charges for the missing signal instead of assuming the best, and the policy engine
refuses to pass a rule it could not actually evaluate.

**The hash is reproducible across languages.** `evidenceHash` is a SHA-256 over a canonical
serialization with ECMAScript number semantics, so TypeScript and Python compute the same
digest for the same manifest. A cross-language conformance test enforces it — the Python
suite reads the TypeScript example and recomputes the hash.

**Test code never runs on the host.** Repository code executes only inside a container with
a read-only root, dropped capabilities, no new privileges, a non-root user, and CPU/memory/
PID limits. If Docker is unavailable, the run reports that it could not test — it does not
fall back to the host.

## Why it's different

| Most AI dev tools | ProofForge |
| --- | --- |
| Generate code and hope | Generate **and** verify, as separate steps |
| Opaque "looks good to me" | Deterministic, **explainable risk score** with published weights |
| A missing signal reads as a clean one | **Unmeasured is never clean** — provenance is part of the spec |
| Run anything, anywhere | **Sandboxed**, no network by default, never on the host |
| Locked to one AI vendor | **Provider-neutral** AI layer |
| No paper trail | Hashable manifest any machine can reproduce |

## Architecture (at a glance)

```text
GitHub App ─┐
Web Dashboard ─→ API (Fastify) ─→ Repository Analyzer (Python)
                    │             Evidence Engine (Python) ─→ Sandbox (Docker)
                    │             Risk Score  ─→  Policy Engine  ─→ verdict
                    └─→ PostgreSQL
```

## Quick start

Requires **Node ≥ 20.11**, **pnpm ≥ 9**, **uv** for the Python services, and **Docker** to
execute tests.

```bash
git clone https://github.com/TheAlphaEngineerCode/proofforge.git
cd proofforge
cp .env.example .env
make setup        # install JS + Python deps
make build        # build all packages
make test         # run the test suite
```

### Produce a manifest from a repository

Builds the sandbox image on first use, runs the repository's tests inside it, and writes an
evidence bundle:

```bash
cd services/evidence-engine
uv run proofforge-evidence build \
  --repo /path/to/repo --owner acme --name demo \
  --url https://github.com/acme/demo \
  --commit <sha> --base <sha> --branch main \
  --output-dir ./bundle
```

### Verify and judge a manifest

```bash
node packages/cli/dist/index.js evidence verify ./bundle/proof-manifest.json
node packages/cli/dist/index.js policy evaluate policies/default.yml ./bundle/proof-manifest.json
```

### Run the storage tests against PostgreSQL

Most tests use the in-memory backend. The Drizzle backend is exercised separately,
because a column constraint or a JSON round-trip is decided by the database, not by
the type checker:

```bash
docker run -d --name pf-pg-test -e POSTGRES_PASSWORD=proofforge \
  -e POSTGRES_DB=proofforge -p 55432:5432 postgres:16-alpine
docker exec -i pf-pg-test psql -U postgres -d proofforge \
  < packages/database/migrations/0000_fine_morbius.sql

TEST_DATABASE_URL=postgresql://postgres:proofforge@localhost:55432/proofforge \
  pnpm --filter @proofforge/database test
```

Without `TEST_DATABASE_URL` those tests skip, so the ordinary suite needs no database.

## Test execution

Tests run in a per-stack container. Only frameworks whose JUnit reporters are built in are
claimed — a repository using anything else is reported as unsupported rather than run with
a command whose failure would be misread.

| Stack | Detected by | Coverage |
| --- | --- | --- |
| pytest | `pyproject.toml` / `requirements.txt` naming pytest | `pytest-cov` |
| vitest | `vitest` in `package.json` | `@vitest/coverage-v8` |

## Commands

| Command | Description | Status |
| --- | --- | --- |
| `proofforge manifest validate <file>` | Validate a manifest against the schema | ✅ |
| `proofforge manifest inspect <file>` | Human-readable summary | ✅ |
| `proofforge evidence verify <file>` | Structure + hash + signature verification | ✅ |
| `proofforge policy validate <file>` | Validate a policy file | ✅ |
| `proofforge policy evaluate <policy> <manifest>` | Judge a manifest against a policy | ✅ |
| `proofforge analyze <path>` | Analyze a repository from the CLI | 🔜 not wired |
| `proofforge init` | Scaffold a policy file | 🔜 not wired |
| `proofforge run --task "..."` | Run an agent task | 🔜 Phase 7 |

The repository analyzer runs as a service today; the `analyze` command has not been wired
to it. See [docs/cli.md](./docs/cli.md) for usage and exit codes.

## The proof-manifest

Every analysis emits a `proof-manifest.json` binding test, security, quality, performance
and operational evidence to a specific commit, protected by a deterministic SHA-256
`evidenceHash` and an optional ed25519 signature. The schema is the single source of truth
(Zod → JSON Schema) — see [docs/evidence-spec.md](./docs/evidence-spec.md).

Quality, performance and operational evidence are currently placeholders: the fields exist
in the spec and the collectors that would fill them do not, so they are reported as
unavailable rather than as zero.

## Roadmap

Foundation and the Evidence Spec ship first; AI agents come last, on top of a base that can
already prove things. Full plan in [ROADMAP.md](./ROADMAP.md).

- **Phase 0** — Foundation (monorepo, infra, CI) ✅
- **Phase 1** — Evidence Spec + CLI ✅
- **Phase 2** — Repository Analyzer ✅
- **Phase 3** — Evidence Engine (sandboxed) ✅
- **Phase 4** — API + Dashboard ✅
- **Phase 5** — GitHub App ✅
- **Phase 6** — Risk & Policy engines ✅
- **Phase 7** — AI agents (provider-neutral) — 🔨 in progress
- **Phase 8** — Distributed workers
- **Phase 9** — SDK & plugins

Phase 7 so far provides the provider interface, cost accounting, containment for untrusted
repository content, and a reviewer agent. No agent is wired into the pipeline yet, and no
agent has been exercised against a live model — the tests script the provider, which covers
the orchestration and says nothing about how a model actually answers.

## The problem

AI can write code faster than any team can review it. But speed without trust is a
liability: a diff that "looks fine" can leak secrets, pull in a vulnerable dependency,
regress performance, or drop an irreversible migration into production. Whether a change
is written by a human or generated by a model, the hard question is the same —

> **Can I trust this change, and can I prove it?**

## The solution

ProofForge treats every change as a **Proof-Carrying Change**: no modification is
considered trustworthy without verifiable technical evidence attached to it.

For any pull request or agent-generated change, ProofForge runs tests, security scans,
dependency and license analysis, performance benchmarks and architectural checks inside
an **isolated sandbox**, computes a **transparent risk score**, applies **configurable
policies**, and emits a signed, hashable **`proof-manifest.json`** that any other machine
can independently verify.

It works in two modes:

- **Validation mode** — point it at an existing pull request; it produces evidence and a
  verdict.
- **Agent mode** — describe a task in natural language; a planning agent proposes a plan,
  an implementation agent writes the code, an **independent reviewer agent** critiques it,
  and the whole change ships with its evidence bundle.

## Why it's different

| Most AI dev tools | ProofForge |
| --- | --- |
| Generate code and hope | Generate **and** verify, as separate steps |
| Opaque "looks good to me" | Deterministic, **explainable risk score** |
| Trust the model | An **independent reviewer agent** must agree |
| Run anything, anywhere | **Sandboxed**, no network by default, never on the host |
| Locked to one AI vendor | **Provider-agnostic** AI layer |
| No paper trail | Signed, hashable manifest reproducible by CLI |

## Quick start

Requires **Node ≥ 20.11**, **pnpm ≥ 9**, and (optionally) **Docker** for local infra.

```bash
git clone <repository>
cd proofforge
cp .env.example .env
make setup        # install JS + Python deps
make build        # build all packages
make test         # run the test suite
```

Bring up local infrastructure (PostgreSQL, Redis, MinIO):

```bash
make docker-up
```

### Verify a manifest with the CLI

```bash
# from the built CLI package
node packages/cli/dist/index.js manifest validate packages/evidence-spec/examples/valid/github-oauth.json
node packages/cli/dist/index.js manifest inspect  packages/evidence-spec/examples/valid/github-oauth.json
node packages/cli/dist/index.js evidence verify   packages/evidence-spec/examples/valid/github-oauth.json
```

## Commands

| Command | Description | Status |
| --- | --- | --- |
| `proofforge manifest validate <file>` | Validate a manifest against the schema | ✅ |
| `proofforge manifest inspect <file>` | Human-readable summary | ✅ |
| `proofforge evidence verify <file>` | Structure + hash + signature verification | ✅ |
| `proofforge analyze <path>` | Analyze a repository | 🔜 Phase 2 |
| `proofforge run --task "..."` | Run an agent task | 🔜 Phase 7 |
| `proofforge policy validate <file>` | Validate a policy file | 🔜 Phase 6 |

See [docs/cli.md](./docs/cli.md) for full usage and exit codes.

## The proof-manifest

Every analysis emits a `proof-manifest.json` binding tests, security, quality,
performance and operational evidence to a specific commit, protected by a deterministic
SHA-256 `evidenceHash` and an optional ed25519 signature. The schema is the single source
of truth (Zod → JSON Schema) — see [docs/evidence-spec.md](./docs/evidence-spec.md).
